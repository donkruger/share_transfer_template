# Selection Persistence

The Selection Persistence feature provides cross-page instrument selection management with business-key deduplication and metadata tracking. It decouples search results from user selections, ensuring that selected instruments persist across multiple search sessions.

## Key Source Files

| File | Role |
|------|------|
| `app/services/selection_manager.py` | `SelectionManager` -- static methods for selection CRUD and metadata |
| `app/components/selection_panel.py` | `SelectionPanel` -- sidebar and main-area selection display |

## Architecture

### Session State Model

`SelectionManager` stores two objects in `st.session_state`:

| Key | Type | Contents |
|-----|------|----------|
| `selected_instruments` | `List[Dict]` | Full instrument data dictionaries for each selected instrument |
| `selection_metadata` | `Dict` | Aggregated metadata: total count, per-instrument timestamps, source queries, last modified timestamp |

The metadata structure:

```
{
  "total_selected": int,
  "selection_timestamps": { instrument_key: ISO timestamp },
  "selection_sources": { instrument_key: search_query_string },
  "last_modified": ISO timestamp or null
}
```

### Business-Key Deduplication

Each instrument is identified by a composite key generated by `get_instrument_key()`:

**Primary key** -- `BUSINESS_KEY|{Exchange}|{Ticker}|{ContractCode}` (all uppercased). This is used when all three components are present.

**Fallback key** -- `LEGACY_KEY|{InstrumentID}|{Name}` (name uppercased). Used when any component of the business key is missing.

This strategy prevents duplicate selections when the same instrument appears in different search results with slightly different metadata, while still supporting instruments that lack standard identifiers.

## API

All methods are `@staticmethod` on the `SelectionManager` class. No instantiation is required.

### Core Operations

| Method | Signature | Returns | Description |
|--------|-----------|---------|-------------|
| `add_instrument` | `(instrument: Dict, source_query: str = "")` | `bool` | Adds instrument if not already selected. Records timestamp and source query. Returns `False` if duplicate. |
| `remove_instrument` | `(instrument: Dict or str)` | `bool` | Removes by instrument dict or key string. Cleans up metadata. Returns `False` if not found. |
| `clear_selections` | `(confirm: bool = False)` | `bool` | Clears all selections. Requires `confirm=True` as a safety gate. |
| `get_selections` | `()` | `List[Dict]` | Returns a copy of all selected instruments. |
| `is_selected` | `(instrument: Dict)` | `bool` | Checks if the instrument's key is already in the selection set. |

### Metadata Operations

| Method | Signature | Returns | Description |
|--------|-----------|---------|-------------|
| `get_selection_summary` | `()` | `Dict` | Returns aggregate stats: total count, unique exchanges, unique asset types, selection sources, oldest/newest selection timestamps. |
| `get_selection_metadata` | `(instrument: Dict)` | `Dict` | Returns per-instrument metadata: `selected_at` timestamp, `source_query`, and `instrument_key`. |
| `get_selection_by_key` | `(instrument_key: str)` | `Dict or None` | Looks up a selected instrument by its computed key. |

## Cross-Page Usage

The `SelectionManager` is imported and used across all pages:

| Page | Usage |
|------|-------|
| **Main Search** (`app/main.py`) | Adds instruments on user click; renders sidebar selection panel; provides source query tracking |
| **AI Assistance** (`app/pages/1_AI_Assistance.py`) | Auto-selects instruments after PDF extraction matching |
| **Portfolio** (`app/pages/2_Portfolio.py`) | Reads selections to generate per-instrument configuration forms; supports individual and bulk removal |
| **Submit** (`app/pages/3_Submit.py`) | Reads selections for review display; supports per-instrument removal; clears all selections after successful submission |

## Selection Panel Component

The `SelectionPanel` component renders in two locations:

1. **Sidebar** -- Compact view showing selected instrument count and names, always visible for context.
2. **Main area** -- Expanded view on the search page, shown below search results when the user has active selections.

Both locations provide instrument removal controls and navigation to the Portfolio page.

## Design Decisions

- **Static methods** -- `SelectionManager` uses only static methods because all state lives in `st.session_state`. No instance state is needed, and this simplifies imports across pages.
- **Copy on read** -- `get_selections()` returns a copy of the list to prevent accidental mutation of session state from consumer code.
- **Confirmation gate** -- `clear_selections()` requires an explicit `confirm=True` to prevent accidental bulk deletion. The UI implements this with a two-click pattern.
- **Source tracking** -- Recording the `source_query` per selection enables the Submit page to show provenance (e.g., "From search: 'Apple'") and supports future analytics.
